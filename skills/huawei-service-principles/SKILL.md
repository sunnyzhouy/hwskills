# Huawei Service Principles

## Overview

Apply an abstract service design quality layer before detailed operating model work. Use this skill to improve closure quality, responsibility clarity, and feedback-loop design.

## Apply Principle Checks First

Judge service design against these questions:

- Is issue intake clear and classifiable?
- Are ownership, escalation, and response clocks explicit?
- Is closure defined by customer resolution rather than internal ticket movement?
- Does the process feed problems back to product, delivery, or management?
- Are service metrics tied to real operational quality?

Read [principle-checklist.md](references/principle-checklist.md) when a formal review is needed.

## Use The Skill In Three Modes

- `Critique`: review an existing service operating model
- `Shape`: improve issue closure and escalation logic before drafting
- `Guardrail`: support `service-itr-execution` while it builds the operating design

## Keep The Abstraction Useful

Explain weaknesses in plain operating language:

- What intake logic is weak
- What escalation path is missing
- What closure definition is fake
- What feedback loop to product or management is absent

## Hand Off Cleanly

If the user needs the actual workflow or operating package:

- Switch to `service-itr-execution` for detailed service workflow and metrics
- Switch to `transformation-pmo-execution` if the goal is organization-wide rollout
