# Framework Map

Use this file when the user needs the relationship between methods rather than a single deliverable.

## Layers

- `Strategy`: BLM and DSTE style work to define direction, choices, control points, and annual priorities
- `Market`: market insight, segmentation, and opportunity selection
- `Product planning`: turn business choices into roadmap, releases, and planning packages
- `IPD execution`: turn approved scope into stage-gated development execution
- `Sales`: connect leads, opportunities, contracts, delivery, and cash realization
- `Service`: connect issue intake, problem solving, escalation, closure, and feedback
- `Transformation`: roll the whole operating system out through PMO, governance, and cadence

## Typical flow

1. Start with strategy if the user is deciding direction or resource focus.
2. Move to product planning when strategic intent must become roadmap and release scope.
3. Move to IPD execution when scope is approved and work must be organized for delivery.
4. Add sales methods when the user is redesigning front-end conversion, opportunity governance, or transaction flow.
5. Add service methods when the user is redesigning post-sales issue handling, escalation, or closed-loop improvement.
6. Add transformation when the user is changing how the organization operates, not just what it plans.

## Diagnostic shortcuts

- If the user asks "what should we bet on", start with strategy.
- If the user asks "what should go into which release", start with product planning.
- If the user asks "how do we execute this across teams", start with IPD execution.
- If the user asks "how do we increase conversion from lead to deal to cash", start with sales.
- If the user asks "how do we close customer issues and feed them back", start with service.
- If the user asks "how do we implement the whole system", add transformation thinking.
